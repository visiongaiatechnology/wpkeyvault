## рҹ”җ What is VGT Key Vault?

WordPress stores API keys in plaintext. Every plugin, every theme, every integration dumps credentials directly into `wp_options` вҖ” readable by anyone with database access.

**VGT Key Vault closes this gap.**

A cryptographic key management system that **seals every API key with AES-256-GCM + AAD Context Binding** before it ever touches the database. Even with full database access, an attacker retrieves nothing but cryptographically worthless ciphertext вҖ” **mathematically bound to the key identifier**.

Built as the cryptographic backbone of the **VisionGaiaTechnology Sentinel ecosystem** вҖ” and now available as a standalone open-source solution for any WordPress installation.

## рҹ”‘ The AAD Context Binding вҖ” What Makes This Different

Standard AES-256-GCM encrypts data. VGT Key Vault goes further with **AAD (Additional Authenticated Data)** вҖ” binding every ciphertext to its identifier:

```
WITHOUT AAD (standard encryption):
  encrypt("sk-groq-xxx") вҶ’ Ciphertext_A
  Attacker copies Ciphertext_A to option "vis_api_key_other"
  decrypt(Ciphertext_A) вҶ’ "sk-groq-xxx" вң“ (Ciphertext Swapping works)

WITH AAD (VGT Key Vault):
  encrypt("sk-groq-xxx", context="vis_api_key_groq") вҶ’ Ciphertext_A
  Attacker copies Ciphertext_A to option "vis_api_key_other"
  decrypt(Ciphertext_A, context="vis_api_key_other") вҶ’ FAIL вң—
  (GCM Authentication Tag mismatch вҖ” manipulation mathematically impossible)
```

**This eliminates an entire class of credential-swapping attacks** that most developers have never even heard of.

## рҹ”‘ Crypto Kernel вҖ” `Crypto_Engine`

```php
// Master Key Derivation via HKDF (not raw salt вҖ” proper key derivation)
hash_hkdf('sha256', SECURE_AUTH_KEY, 0, 'vgt_vault_master_domain', AUTH_SALT);

// Encryption with AAD Context Binding
Crypto_Engine::encrypt($api_key, $option_name);

// Decryption вҖ” verified against AAD (tamper detection built-in)
Crypto_Engine::decrypt($ciphertext, $option_name);
```

**Key Architecture:**
- Master key derived via **HKDF-SHA256** from WordPress salts вҖ” never stored
- Every encryption uses a **fresh random IV** (`random_bytes`)
- **GCM Authentication Tag** appended вҖ” detects any modification
- **AAD Context ID** binds ciphertext to its exact storage location

## рҹ”Ң Inter-Plugin API вҖ” One Line Access

Other plugins in your ecosystem retrieve keys with a single authenticated call:

```php
use VGT\Vault\API;

// O(1) retrieval + AES-256-GCM decryption in one call
$api_key = API::get_key('vis_api_key_groq');

// Throws RuntimeException if key missing or tampered
try {
    $key = API::get_key('vis_api_key_stripe');
} catch (\RuntimeException $e) {
    // Handle missing/compromised key
}
```

**No plaintext ever stored. No raw option access. One secure interface.**

## вҡ пёҸ Important Security Notice

```
вҡ пёҸ  VGT Key Vault derives its master key from WordPress salts.

    If you change AUTH_SALT or SECURE_AUTH_KEY in wp-config.php,
    ALL stored ciphertexts become permanently unreadable.

    Before migration or salt rotation:
    1. Decrypt and export all keys from the Vault Dashboard
    2. Rotate salts
    3. Re-import keys into the new Vault
```
